
# ביקורת מקיפה – מערכת גמרא SMS

---

## 🎯 מה המערכת עושה?

מערכת SMS ללימוד גמרא יומי. משתמשים נרשמים דרך SMS, בוחרים מסכת ודפים, ומקבלים שאלות חזרה יומיות לפי הקצב שהגדירו. המערכת מנוהלת דרך ממשק Flask עם דשבורד, סימולטור, ועורך תבניות.

---

## ✅ מה עובד טוב

### ארכיטקטורה כללית
- **הפרדת אחריות סבירה**: כל קובץ אחראי על תחום ברור – `database.py`, `registration.py`, `scheduler.py`, `questions_engine.py`, `sms_service.py`, `simulation_system.py`.
- **תמיכה כפולה SQLite/PostgreSQL**: המערכת תומכת בשני מנועי DB, מה שמאפשר פיתוח מקומי ו-deploy ב-Heroku/Render.
- **סימולטור מובנה**: ממשק ה-Web לסימולציה הוא רעיון מצוין לפיתוח ובדיקות.
- **תבניות SMS ניתנות לעריכה**: הפרדת הטקסטים ל-`sms_templates.json` היא גישה נכונה.
- **Scraper לתוכן**: כלי לאיסוף שאלות מ-dafyomi.co.il הוא חכם ומאפשר הרחבה.

## 🔴 בעיות קריטיות

### 1. **סוד חשוף ב-`.env` ב-Git**
```
INFORU_TOKEN=3a45cda4-ff7e-4b99-91c8-d96e74cb872b
```
ה-`.env` עם ה-API token האמיתי נמצא בריפוזיטורי! זה **חור אבטחה חמור**. כל מי שיש לו גישה ל-repo יכול לשלוח SMS על חשבונך.
**פתרון**: הוסף `.env` ל-`.gitignore` מיידית, החלף את ה-token, ועבוד עם `.env.example`.

### 2. **`LIVE_MODE` הוא משתנה גלובלי בזיכרון**
```python
LIVE_MODE = False  # sms_service.py
```
בסביבת production עם מספר workers (Gunicorn), כל worker יחזיק ערך שונה. הפעלת LIVE_MODE בדשבורד לא תשפיע על כל ה-workers.
**פתרון**: שמור את המצב ב-DB או ב-Redis.

### 3. **`USER_STATES` הוא dict גלובלי בזיכרון**
```python
USER_STATES = {}  # simulation_system.py
```
אותה בעיה – מצב שיחה של משתמש יאבד בכל restart, ולא יעבוד עם מספר workers.
**פתרון**: שמור states ב-DB (טבלת `user_states`).

### 4. **Scheduler בתוך Flask Thread**
```python
thread = threading.Thread(target=scheduler_loop, daemon=True)
```
הרצת scheduler כ-daemon thread בתוך Flask היא פתרון שביר. ב-Gunicorn עם `--workers=2` יפעלו שני schedulers במקביל ויכפילו שליחות.
**פתרון**: השתמש ב-APScheduler, Celery Beat, או Heroku Scheduler כ-dyno נפרד.

## 🟠 בעיות ארכיטקטורה משמעותיות

### 5. **`PostgresConnWrapper` – wrapper שביר ומסוכן**
```python
def execute(self, query, params=None):
    if "last_insert_rowid()" in query:
        cur = self.conn.cursor()
        if "users" in self._last_table:
            cur.execute(f"SELECT id FROM {self._last_table} ORDER BY id DESC LIMIT 1")
```
זה פתרון מסוכן מאוד – `_last_table` הוא class variable (לא instance variable!), כלומר משותף לכל ה-connections. בסביבה concurrent זה יגרום לבאגים קשים לאיתור.
**פתרון**: השתמש ב-`RETURNING id` ב-PostgreSQL, או SQLAlchemy שמטפל בזה אוטומטית.

### 6. **פתיחה וסגירה של DB connection בכל פעולה**
```python
conn = get_conn()
# ...
conn.close()
```
כל פונקציה פותחת ומסגרת connection בנפרד. זה יוצר overhead גדול ומסכן ב-race conditions.
**פתרון**: Connection pooling (SQLAlchemy + `scoped_session`).

### 7. **קריאת `sms_templates.json` מהדיסק בכל הודעה**
```python
def get_template(template_name_pos=None, **kwargs):
    with open(template_path, "r", encoding="utf-8") as f:
        templates = json.load(f)
```
כל שליחת SMS קוראת את הקובץ מהדיסק מחדש. עם 1000 משתמשים זה יהיה צוואר בקבוק.
**פתרון**: Cache בזיכרון עם invalidation בעת עדכון.

### 8. **הרשמה דרך SMS בפורמט קשיח מדי**
```
הרשמה, משה, כהן, בני ברק, 25, ברכות, ב ע"א עד י ע"ב, 1, 18
```
9 פרמטרים בהודעה אחת, כולל גימטריה עברית, עם פסיקים. כל שגיאה קטנה תכשיל את ההרשמה. זה לא UX ריאלי לאנשים שאינם טכניים.
**פתרון**: תהליך הרשמה שלב-אחר-שלב (conversational flow) כמו שכבר קיים ל-update_daf.

### 9. **שאלות באנגלית בלבד**
```json
"question_en": "(a) What is the earliest time to recite Keri'as Sh'ma..."
"question_he": ""
```
כל השאלות ב-`berachos_questions.json` הן באנגלית, ושדה `question_he` ריק. המערכת מיועדת לקהל דובר עברית.
**פתרון**: תרגום השאלות (ה-scraper כבר מציין `python translate_questions.py` שלא קיים).

## 🟡 בעיות יעילות וקוד

### 10. **SQL ישיר בכל מקום – אין ORM**
שאילתות SQL מפוזרות ב-5+ קבצים שונים. שינוי schema דורש חיפוש ידני בכל הקוד.
**פתרון**: SQLAlchemy ORM או לפחות ריכוז כל ה-queries ב-`database.py`.

### 11. **אין validation על קלט משתמש**
```python
phone = request.args.get('Phone') or request.form.get('Phone')
```
אין sanitization על מספר טלפון, אין הגנה מ-SQL injection (אמנם parameterized queries, אבל אין validation).

### 12. **`automated_test.py` שבור**
```python
questions = select_questions_for_range(
    t['id'],        # ← signature שגויה!
    daf, rate, count=2, already_sent=[]
)
```
ה-signature של `select_questions_for_range` מקבלת `(questions: list, start_f, end_f, already_sent_ids)` אבל הטסט קורא לה עם `(tractate_id, daf, rate, count, already_sent)`. הטסט לא עובד.

### 13. **`reverse_hebrew_line` – פונקציה ריקה עם 15 שורות הערות**
```python
def reverse_hebrew_line(line: str) -> str:
    return line  # לא עושה כלום
```
קוד מת שמבלבל.

### 14. **`dj_database_url` מיובא אך לא בשימוש**
```python
import dj_database_url  # database.py שורה 10
```
הספרייה מותקנת ומיובאת אבל לא נקראת בשום מקום בקוד.

### 15. **אין logging מסודר**
כל ה-logging הוא `print()`. בסביבת production אין יכולת לסנן לפי רמת חומרה, לשמור לקובץ, או לשלוח ל-Sentry/Datadog.

11. **הוסף גרפים לסטטיסטיקות** (תגובות, פעילות לפי שעה)

## סיכום

המערכת היא **proof-of-concept מרשים** עם רעיון מצוין ומימוש בסיסי עובד. הארכיטקטורה הכללית הגיונית, אבל יש מספר בעיות שיש לטפל בהן לפני שימוש ב-production: בעיקר חשיפת ה-token, ה-global state שלא יעבוד עם מספר workers, וה-scheduler שעלול לשלוח הודעות כפולות. עם תיקון הבעיות הקריטיות ושיפור ה-UX של ההרשמה, המערכת יכולה להיות כלי לימוד SMS אמיתי ושימושי.
