# מסמך אפיון משימה למודל AI: שיפור חוויית משתמש (UX) למערכת מרובת-מנויים

## רקע ומטרת המשימה
המערכת תומכת בריבוי מנויים לאותו משתמש (למשל: לימוד מספר מסכתות במקביל, או מספר טווחי דפים באותה מסכת). 
המטרה של משימה זו היא לשפר את חוויית המשתמש כך שהמערכת תתנהג בצורה טבעית ופשוטה (כמו במנוי יחיד) עבור רוב המשתמשים, ותחשוף מורכבות (תפריטי בחירה) רק כאשר המשתמש מנהל בפועל מספר מנויים. כמו כן, יש לנהל בצורה חכמה תור של שאלות שמוגדרות לאותה שעה, ולספק משוב מסכם בסיום לימוד יומי.

## פירוט הדרישות (Requirements)

### 1. ניהול פעולות (עדכון דף, שעה, הקפאה, ביטול)
- **מצב נוכחי:** המשתמש נדרש להזין קלט משולב (למשל: `1 25` - בחירת מנוי + ערך).
- **הדרישה החדשה:**
  - כאשר מתקבלת בקשה לפעולה, המערכת תבדוק את מספר המנויים הפעילים של המשתמש.
  - **אם קיים מנוי אחד בלבד:** המערכת תבקש את הנתון ישירות (למשל "לאיזה דף תרצה לעדכן?") ללא ציון מספרי מנויים, ותבצע את הפעולה על המנוי היחיד.
  - **אם קיימים מספר מנויים:** המערכת תציג תפריט של המנויים **כולל טווח הדפים** (למשל: "1. ברכות (ב-ה)"). המשתמש ישיב רק עם המספר. לאחר מכן המערכת תבקש את הנתון החדש.

### 2. בקשת שאלות יזומה ("שלח שאלה")
- **אם למשתמש מנוי אחד:** שליחת השאלה באופן מיידי.
- **אם למשתמש מספר מנויים:** המערכת תציג תפריט ותשאל: "יש לך מספר מסלולי לימוד פעילים. באיזה מסלול תרצה להיבחן כעת?" עם רשימת המנויים (כולל טווחי דפים). לאחר הבחירה, תישלח השאלה.

### 3. ניהול תור שאלות מתוזמנות (Scheduler Queue)
- **הבעיה:** מספר מנויים שמתוזמנים לאותה שעה מייצרים הצפה של שאלות באותו זמן.
- **הפתרון (ניהול Queue):**
  - ה-Scheduler יזהה שיש מספר שאלות ממתינות לאותה שעה לאותו משתמש.
  - במקום לשלוח את כולן, המערכת תשלח הודעת "תפריט תור": 
    *"הגיע הזמן ללימוד! יש לך שאלות ממתינות. מאיזו מסכת תרצה להתחיל? \n 1. ברכות (ב-ה) \n 2. שבת (כ-ל)"*
  - המשתמש יבחר מספר, והמערכת תשלח את השאלה המתאימה.
  - לאחר שהמשתמש עונה נכונה (ומסיים את השאלות לדף הנוכחי במסכת זו):
    - המערכת תבדוק כמה מנויים נותרו בתור לאותה שעה.
    - **אם נותרו >= 2 מנויים:** המערכת תשלח משוב חיובי + "נותרו לך שאלות. במה נמשיך?" עם תפריט של הנותרים.
    - **אם נותר מנוי 1:** המערכת תשלח משוב חיובי + "מצוין! סיימת את [מסכת א']. ממשיכים למסכת האחרונה - [מסכת ב' + טווח]:" ומיד תצרף את השאלה.
    - **אם לא נותרו מנויים:** המערכת תעבור לסטטוס סיום הלימוד היומי (סעיף 4).

### 4. הודעת משוב / סגירת מעגל בסיום לימוד יומי
- כאשר המשתמש מסיים לענות על כל השאלות היומיות עבור מנוי מסוים, וללא שאלות ממתינות בתור, המערכת תשלח הודעת "סגירת מעגל".
- **המידע הנדרש בהודעה:**
  - ברכה על הסיום ("כל הכבוד!").
  - ציון המסכת וטווח הדפים שנסתיים כעת.
  - הצצה למחר: מהו הדף הבא בתור ללימוד במנוי זה.
  - תזכורת לשעת השליחה המוגדרת למחר.
- **פורמט לדוגמה:**
  *"תשובה נכונה, כל הכבוד!  סיימת את הלימוד להיום במסכת ברכות (ב-ה). הלימוד מחר יהיה דף ו', והשאלות ישלחו בשעה 18:00."*

## הערות טכניות למיישם
1. **State Machine:** נדרש להרחיב או לשנות את ניהול המצבים (`USER_STATES` או ב-DB) כדי לתמוך במצבים החדשים: `WAITING_FOR_SUB_SELECTION_QUESTION_QUEUE`, `WAITING_FOR_SUB_SELECTION_ACTION` וכדומה.
2. **חישוב דף הבא:** לצורך הפקת ההודעה בסעיף 4, יש לשלוף את נתוני המסכת (מתוך ה-JSON הרלוונטי או לוגיקת המעבר) כדי להציג איזה דף מגיע אחרי הדף הנוכחי, בהתאם לטווח המוגדר.
3. **עיצוב תפריטים:** בכל פעם שמוצג תפריט מנויים, חובה לכלול את טווח הדפים הרלוונטי כדי למנוע בלבול (למקרה של משתמש שלומד את אותה מסכת בשני טווחים שונים).
